처음 블로그를 만들면 글의 내용보다도 형식이 더 어렵게 느껴질 때가 많습니다. 제목은 어떻게 나누고, 코드는 어떻게 보여주고, 링크는 어떤 방식으로 정리해야 읽는 사람이 덜 피곤할지 고민하게 됩니다. 이 글은 그런 고민을 줄이기 위해 자주 쓰는 마크다운 요소를 한 번에 정리한 실전용 예시입니다. 실제 포스트를 작성할 때 복사해서 시작점으로 쓰기 좋게 구성했고, 한국어 본문에서 자주 부딪히는 줄바꿈, 강조, 코드 블록, 표, 인용문, 체크리스트까지 모두 담았습니다. 문법을 외우는 것보다 중요한 것은 읽는 흐름을 만드는 일입니다. 그래서 이 글에서는 문법 그 자체보다, 문법을 어떤 문맥에서 쓰면 좋은지까지 함께 설명합니다. 길게 보이더라도 한 번만 끝까지 읽고 나면 이후 글쓰기 속도가 꽤 빨라질 것입니다.

## 문서 구조 먼저 잡기

좋은 포스트는 대부분 구조가 먼저 정리되어 있습니다. 보통은 `h2`를 큰 챕터로 두고, 그 안에서 `h3`로 세부 문단을 나누는 방식이 가장 안정적입니다. 이 방식은 TOC를 자동으로 만드는 테마에서도 잘 동작하고, 글이 길어질수록 독자가 현재 위치를 파악하기 쉬워집니다.

### 초안 작성 순서

1. 제목과 핵심 메시지를 먼저 한 줄로 적습니다.
2. `h2` 기준으로 3~5개 챕터를 만든 뒤 핵심 문장을 배치합니다.
3. 각 챕터를 `h3`로 세분화하고 예시를 추가합니다.

### 체크리스트 방식

- 작성 전: 전달할 대상 독자를 한 문장으로 정의하기
- 작성 중: 한 문단 한 메시지 원칙 유지하기
- 작성 후: 소리 내어 읽으며 어색한 연결어 제거하기

## 텍스트 강조와 링크

강조는 많을수록 좋은 것이 아닙니다. **정말 중요한 문장**만 굵게 처리하고, *보조적인 뉘앙스*는 이탤릭으로 제한하는 편이 읽기 편합니다. 이미 중요하지 않은 내용까지 강조하면 정보 우선순위가 흐려집니다. 더 이상 유효하지 않은 내용은 ~~취소선~~으로 처리하면 수정 이력이 명확해집니다.

내부 문서 연결은 블로그에서 매우 중요합니다. 예를 들어 [소개 페이지]({{ '/about/' | relative_url }})처럼 `relative_url` 필터를 사용하면 배포 경로가 바뀌어도 링크가 안정적으로 유지됩니다. 외부 참고 문서는 [Markdown Guide](https://www.markdownguide.org/)처럼 명확한 텍스트로 연결하는 것이 좋습니다.

### 키보드 단축키 표기

편집기 사용 팁을 쓸 때는 <kbd>Ctrl</kbd> + <kbd>S</kbd>, <kbd>Ctrl</kbd> + <kbd>Shift</kbd> + <kbd>P</kbd>처럼 `kbd` 태그를 쓰면 가독성이 좋아집니다.

### Liquid 문법을 글자 그대로 보여주기

Liquid 코드를 설명할 때는 아래처럼 `{% raw %}{% raw %}` 블록으로 감싸야 실제 문법이 출력됩니다.

{% raw %}
```liquid
<a href="{{ '/archive/' | relative_url }}">아카이브</a>
```
{% endraw %}

## 코드 블록 사용 패턴

코드는 언어를 명시해야 하이라이팅이 정확합니다. 실행 순서가 있는 명령은 bash로 묶고, 개념 설명은 Python이나 JavaScript를 분리해서 보여주는 것이 좋습니다.

### Python 예시

```python
def reading_time_korean(text: str, wpm: int = 350) -> int:
    words = len(text.split())
    minutes = max(1, round(words / wpm))
    return minutes

sample = "이 문장은 읽기 시간 계산 예시입니다."
print(reading_time_korean(sample))
```

### JavaScript 예시

```javascript
const toggleTheme = () => {
  const root = document.documentElement;
  const next = root.dataset.theme === "dark" ? "light" : "dark";
  root.dataset.theme = next;
  localStorage.setItem("theme", next);
};

document.querySelector(".theme-toggle")?.addEventListener("click", toggleTheme);
```

### Bash 예시

```bash
bundle install
bundle exec jekyll serve --livereload
```

인라인 코드는 `bundle exec jekyll build`처럼 문장 안에서 짧게 보여줄 때 유용합니다.

## 표, 인용문, 리스트 조합

표는 비교가 필요한 정보에 가장 효과적입니다. 특히 작성 규칙처럼 반복 확인이 필요한 항목을 정리할 때 시간을 많이 아껴 줍니다.

| 항목 | 권장 방식 | 이유 |
| --- | --- | --- |
| 제목 길이 | 20~40자 | 검색/목록에서 가독성 확보 |
| 문단 길이 | 3~5문장 | 모바일에서 피로도 감소 |
| 코드 블록 | 언어 지정 필수 | 하이라이팅 정확도 향상 |

> 글쓰기는 지식을 뽐내는 행위가 아니라, 다음의 나를 돕는 정리 작업이다.

중첩 리스트는 절차 설명에서 특히 유용합니다.

- 글 발행 전 점검
  1. 맞춤법 검사 실행
  2. 링크 유효성 확인
  3. 대표 이미지 확인
- 발행 후 점검
  1. 검색 페이지 반영 확인
  2. 카테고리/태그 페이지 노출 확인
  3. 모바일 레이아웃 재검토

## 마무리

마크다운은 문법 자체보다 습관이 더 중요합니다. 처음에는 요소를 많이 써 보되, 반복하면서 본인만의 템플릿을 줄여 가는 방식이 가장 효율적입니다. 예를 들어 기술 글이라면 문제 정의, 시도, 실패 원인, 해결 방법, 검증 결과, 회고라는 기본 뼈대를 정해 두고 매번 같은 순서로 작성해 보세요. 그러면 글의 완성도가 안정적으로 올라가고, 작성 시간도 예측 가능해집니다. 또한 블로그 운영 관점에서는 한 편의 완벽한 글보다, 일정한 품질의 글을 꾸준히 발행하는 편이 훨씬 큰 효과를 만듭니다. 오늘은 이 가이드를 기반으로 짧은 글 하나를 직접 작성해 보고, 다음 글에서 표와 코드 블록을 자연스럽게 섞어 보는 것을 추천합니다. 작은 반복이 쌓이면 문서화 역량은 분명히 체감될 만큼 좋아집니다.
